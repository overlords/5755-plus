# 平台余额/余额支付列范围外·可拓展;v2 不自持玩家余额、走持牌机构直连代收

## 背景

#58 支付集成定为「平台自建收银台 + 微信/支付宝直连」。期间有人提出「内部支付 / 余额支付」——玩家先向 5755 主账户预存真实货币(¥),下单时从该余额扣减、资金不出平台。经 grill(2026-06-14)厘清:提出方明确「**余额即平台币**」。本 ADR 记录对它的定位与否决理由。

## 决定

**v2 不自持玩家余额。** 余额支付(= 平台币体系)定为 `01 §5` 评审门后的「**范围外·可拓展项**」——门开着、非永久排除,但 **v2 内不做、不阻塞 GA**。支付资金一律走持牌机构(微信/支付宝)**直连代收**,平台账上不留沉淀。

未否决「将来纳入」,但重新评估须同时满足:(1) 产品确有余额需求;(2) 法务/财务给出合规路径结论(申牌 vs 单用途备案);(3) 走完 `01 §5` 能力白名单/公开 API/Manifest/依赖/接入自检评审。**不得由实现侧把它当「#58 新增一种付款方式」直接落地。**

完整需求与代价规格(用户故事 / 服务端子域 / 口径变更 / 测试 / 合规清单)立为 **#62**(`ready-for-human`);在上述三条件全满足前保持 `ready-for-human`、不流转 `ready-for-agent`,届时由 #62 进入实现评审。

## 为什么(合规定性)

玩家预存的真实货币是平台对玩家的**预收款负债 + 沉淀资金**(「还没花出去、仍是钱的钱」),与「平台币」(已换成游戏内权益、单向消耗不可逆兑现的钱)本质同族,但**合规分量更重**:它把平台从「游戏运营者」拽进《非银行支付机构监督管理条例》的「储值账户运营」——备付金 100% 集中存管、**二清风险**、支付牌照(注册资本 1 亿元起 + 储值类再加 1 亿)。游戏分发平台天然要给第三方 CP 分账,极易越过「单用途预付」线滑向二清/多用途。`01 §4.2` 排除「平台币」所守护的「不沾可逆现金化的真实货币购买力」这条精神,同样、甚至更强地覆盖余额。

旧实现佐证「全新业务、无可迁移」:旧 SDK(`~/Developer/m5755`)`PaymentMethod` 只有 `SDK_CONTAINER`、`User` 仅 `token`+`account`,支付靠服务端下发 `paymentUrl` 跳转;旧平台(`~/Developer/U10`)20 个 model、10 个 migration 零余额字段,支付链路全程不读不扣余额。

## 前向兼容缝(关键,几乎零成本)

#58/#60/#61 须保持「**支付方式对 SDK 与 04 契约不可知**」:SDK 只建**方式无关的 `Order`**、拉起平台收银台、收 3 态客户端支付回调;「用什么付」活在平台收银台 H5 + 服务端。**别把「微信/支付宝」当枚举渠道写进 04 契约或 `Order`。** 旧 SDK「服务端下发 `paymentUrl`、SDK 只跳转」已是此基因。

如此,将来若评审通过纳入余额,**client 侧(AAR / 公开 API / 给游戏的 04 契约)几乎零破坏**,代价集中在:
- 服务端**净新增子域**:余额账本 + 流水 + 预存入账(仍走微信/支付宝)+ 原子扣减 + 退款/对账;
- **3 处口径编辑**:`02` 的「充值」三重重载切分(预存入账 / 余额扣减 / 发货通知)、`05 §3.2` 充值回调语义放宽(「确认第三方支付结果」→「确认本笔订单资金到位」)、`01 §4.2`+`06a §8` 平台币禁词/范围外解禁;
- **合规前置**:申牌 vs 单用途口径、备付金/二清规避(持牌分账直连)、未成年人限额前移到「预存」动作、退款可退性、反洗钱、预收款会计税务。

## 考虑过的其他选项

- **(a) 现在做进 #58** — 否决:跨穿全部支付/账户口径(3 处口径硬伤)+ 退款对账新子域 + 未解决的合规定性,GA 受控验收窗口内不现实,且顶撞「最小化 AAR」立项身份。
- **(b) 永久排除** — 未选:提出方想保留「可拓展」,不愿把门关死。
- **(c) 范围外·可拓展 + 锁前向兼容缝 + 本 ADR** — **选中**。

## 后果

GA 终点线 6 项不受影响。`#60`/`#61` 带「支付方式不可知」验收约束(零额外成本)。真要做时按本 ADR 的代价清单走 `01 §5` 评审门 + 合规前置,合规/财务/法务部分明确移交对应职能,不在 SDK/契约技术范围内消化。
