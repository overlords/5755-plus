# 渠道标识符防篡改:v2 不上密码学防护,冻结扩展形态留 #32

## 背景

channelId 写在 APK Signing Block 自定义块(`0x71777777`),`channel-pack` 免重签写入(walle 方案:v2/v3 签名不覆盖 signing block 自定义条目)。**免重签是 #32 落地页动态打包的硬前提**——落地页要为每次点击 / 每个渠道实时盖章,重签需私钥且慢,实时重签不可行。

但免重签的另一面:**signing block 自定义块不被签名覆盖 = 任何持包者可改写** → 渠道块零完整性保护。`01 §6` 此前还把 `operations_mismatch`(与运营配置不一致 → default)写成与已实现 reason 并列,实则 `ChannelRules` 无此路径、服务端无渠道注册表(已另行标注为预留)。

本 ADR 经 grill(2026-06-15)产出,记录:渠道防篡改在 v2 做到哪、未来扩展形态如何冻结、为何不现在上密码学方案。

## 决定

1. **v2 不实现任何密码学防篡改**(不上 HMAC 签名、不上 clickid 点击归因、不上服务端渠道注册表校验)。GA 阶段 channel 实发 `default`、无真实渠道流量,此缺口不构成 GA 损失。防篡改深度实现**留 #32 生成端立项**。

2. **冻结 `channelId` 字段语义**:纯渠道标识符、解析后值(`04 §1.6`),格式 1-64 小写 `[a-z0-9_.-]`(`01 §6`)。未来防篡改签名(`channelSig` / HMAC)与点击归因(clickid)**各走独立的 APK Signing Block 自定义块 + 独立上报字段**,不复用、不挤占 channelId 的字段 / 格式 / 块。

3. **防篡改范式 = 服务端验签,不靠载体完整性**。免重签下 signing block 块本就可改,防护只能靠「块内容带平台签名、服务端验真」——攻击者改得了块、却伪造不出有效签名(无平台密钥)。这是免重签场景下**唯一**可行的防篡改范式;靠「载体不可改」的方案与免重签直接对立、不可用。

4. **`operations_mismatch` 归服务端判定**。渠道注册表在服务端,SDK 无注册表、永不产出此 reason。未来落地时:要么 `config` 下发渠道清单给 SDK 本地校验,要么明确为服务端侧判定(不在 SDK 诊断五字段)。`01 §6` 已标注为预留未实现。

5. **免重签硬约束**:① 包须**已 v2/v3 签名**才存在 signing block 可写,纯 v1 包无 signing block、方案不适用(`channel-pack` 已要求);② minSdk 21 → 走 **v1 + v2/v3 混合签名**(v1 给 API 21-23,v2/v3 给 24+),渠道块写 signing block 时 v1(不覆盖它)与 v2/v3(排除它自身)**都不破坏**;③ SDK 运行时**自解析 signing block** 读渠道 / 扩展块,独立于系统用哪个方案验签——故 API 21-23(系统只验 v1)设备 SDK 照样读得到。

6. **生成端三硬约束**(#32 实现,本 ADR + `02` 钉口径):channelId **禁用保留值 `default`**;**全局唯一**(跨会长不得撞同名,否则归因混淆);生成端格式**与读取端归一化对齐**(同样 1-64 小写 `[a-z0-9_.-]`),否则大小写 / 非法字符经读取端归一后撞车或丢归因。

## 为什么(威胁模型 + 选型理由)

威胁模型(按实际收益排序):

| 攻击 | 手法 | 收益 | 谁能防 |
| --- | --- | --- | --- |
| **W1 刷量** | 改渠道刷安装 | **被 #32 决议5「分成只认充值订单」废掉**(假安装无充值=无分成) | 业务(充值门槛) |
| **W1' 贴牌截流** | 把**合法**渠道包投到 SEO / 冒充官方下载站,截自然付费量 | 偷自然量分成,但执行门槛高、可对账监测 | **仅 clickid**(包签名防不住) |
| **W2 伪造未注册渠道** | 凭空造 `guild_FAKE` | 结算无对应会长主体、拿不到钱 → 低 | 注册表 / HMAC |
| **W3 破坏成 default** | 删 / 改渠道块 | 免重签必失守,但攻击者自己无收益 → 低 | 无(免重签固有) |

关键判断:

- **W1' 是唯一有真实平台损失、却被密码学防不住的攻击**。它用的是平台签发的**合法**渠道块、免重签复制到**官方原始包**(同包名 / 证书 / 版本),HMAC 验签通过——因为会长推广的就是这个游戏、包标识天然匹配。W1' 的作弊在**分发行为**(把合法包投到截流场景),不在 APK 篡改;包内签名无法区分分发路径。唯一能防它的是 **clickid 点击归因**(一次点击一个 ID,复制包 = 同 clickid 多安装 = 异常),属 #32 二期。
- **HMAC 绑 APK 真正买到的是「防跨游戏搬运」**(把 `guild_A` 块贴到别的游戏包),不是 W1'。
- 真正变现的 W1 / W1' 已被业务侧(充值门槛 + 零摩擦落地页消除自造动机 + 分成对账监测 + 决议6 邀请制审核)压到低敞口。故 v2 不上密码学,靠业务兜底 + 未来注册表(#32)挡伪造、clickid(二期)防 W1'。
- 前向兼容审查:**注册表**预留位够(服务端加表即可);**HMAC** 是硬卡点(当前 channelId 格式装不下签名、上报只传解析后值);**clickid** 与 channelId 正交、非破坏式可加。所以正确的「现在该做的」不是改代码,而是 ADR 冻结扩展形态(独立块 / 独立字段 / 服务端验签 / 免重签兼容),让三方案将来都能干净接入。

## 考虑过的其他选项

- **(a) v2 现在上 HMAC 绑 APK** — 否决:防不住 W1'(合法块 + 官方包静态一致),只防 W2 / 跨游戏搬运;且当前 channelId 格式 / 上报装不下签名、必破坏性改契约;W1' 收益本就低,性价比差。
- **(b) 扩大 channelId 格式装签名串** — 否决:污染 channelId 归一化语义(小写 / 字符集),让「渠道值」与「签名」混在一字段;签名应走独立块 / 字段。
- **(c) v2 现在落服务端注册表校验(operations_mismatch)** — 未选:注册表填充靠 #32 推广链接,v2 先落空表 = 所有真渠道被拒;GA 无真实渠道流量、无紧迫性。留 #32 与推广链接注册一并落。
- **(d) clickid 提前到 v2** — 否决:最重(每点击打包 + 服务端点击 / 安装比对),W1' 收益低,#32 二期增强足够。
- **(e) v2 不实现 + ADR 冻结扩展形态 + 生成端硬约束** — **选中**:零代码改动,扩展位形态钉死,三方案将来都能干净接入、不破坏现有 channelId 链路。

## 后果

- **v2 不动代码**。`01 §6` 的 `operations_mismatch` 预留标注 + 渠道防篡改表下注另由同批 PR 落;`02` 术语补生成端三硬约束;#32 钉防篡改设计点 + 本 ADR 链接。
- **channelId 字段语义冻结**:任何未来防篡改 / 归因增强不得挤占 channelId 字段或扩大其格式。
- **双源分工明确**:`manifest` 源 = 接入方构建期写死的静态渠道(自有签名、重签无所谓);`signing_block` 源 = 平台落地页**动态免重签**盖的渠道(#32 推广链接)。两源非冗余,是静态 vs 动态两类来源。
- **#32 立项时**:实现 `operations_mismatch`(服务端注册表)+ 决定是否上 HMAC(绑包名防跨游戏搬运)/ clickid(二期防 W1');所有扩展走独立 signing block 块 + 独立上报字段 + 服务端验签,遵免重签 + v1/v2/v3 混合签名约束;方案最终选型(签名 vs 注册表 vs clickid 叠加)在 #32 立项时另落 ADR。
