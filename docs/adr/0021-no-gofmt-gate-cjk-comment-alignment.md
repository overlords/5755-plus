# CI 不设 gofmt 门禁,保护中文行尾注释的手工对齐

状态:已接受(2026-06-16)

## 背景

Go 生态的默认卫生是「gofmt-clean」,多数项目会在 CI 加 `gofmt -l` 门禁(有未格式化文件即 fail)。

本仓代码注释**全为中文**(CLAUDE.md「所有文档与代码命名中文」),且普遍**按显示宽度手工对齐**行尾注释以提升可读性,例如:

```go
PaymentPending = "待支付"  // 订单已建,尚未支付
PaymentPaid    = "已支付"  // 渠道确认收款
PaymentFailed  = "支付失败" // 支付明确失败
```

`gofmt` 用 `tabwriter` 按 **rune 数**(非显示宽度)对齐行尾注释:CJK 字符各算 1 rune,但编辑器渲染为双宽,于是 gofmt「修复」后的中文注释列在编辑器里**视觉错位、更难读**。一次 CI 健全度 grill 实测:`gofmt -l server/` 报 4 个文件不干净,**4/4 全是这种中文注释对齐漂移**,无一是真正的格式 bug。

## 决定

**CI 不加 `gofmt -l` 门禁**;`server` job 保持仅 `go vet`。格式由开发者自律(可选 gofmt-on-save,但**不强制**,允许为可读性手工对齐中文行尾注释)。

正确性网由其余门禁兜住:`go vet` + 真实 Postgres 的 HTTP-seam 全套测试 + `-race`(并发竞争)+ 生产排除子集 + Android 纯净门禁。gofmt 关切的是**美观一致**,不是正确性。

`ci.yml` 的 `go vet` 步附指针注释,指向本 ADR,避免被「顺手加 gofmt」推翻。

## 为什么

- **收益 < 成本(本项目特定)**:gofmt 门禁的收益是纯美观一致;成本是损中文注释可读性,与「中文优先」ethos 直接冲突。这是少见的「gofmt 该让位」情形。
- **正确性不靠 gofmt**:vet/test/race 已覆盖正确性类问题;格式漂移是 cosmetic、不影响行为。
- **难逆性**:一旦有人 `gofmt -w` 全仓,所有手工中文对齐一次性丢失、不可批量还原;故需显式立档止住「反射性加 gofmt」。

## 考虑过的其他选项

- **加 `gofmt -l` 门禁(gofmt 当家)** — 否决:要重排现有 4 文件并禁止以后手工对齐中文注释,gofmt 的 rune 对齐使中文列视觉错位、损可读性。
- **上 golangci-lint** — 否决(同向):加工具 + 配置 + 大量存量告警 triage,与最小依赖拉扯;且其 `gofmt`/`gofumpt` 子 linter 会带回同一中文对齐问题。深静态分析可作为独立议题,不在「健全」必需集。
- **保持仅 `go vet`** — **采纳**:守正确性、不碰美观、不损中文可读性。

## 后果

- `ci.yml` 维持 `server` job 仅 `go vet`(无 gofmt 门禁);`go vet` 步加指向本 ADR 的注释。
- 若未来团队决定改向「gofmt 当家、放弃手工中文对齐」,须先改本 ADR 状态、再全仓 `gofmt -w` 并加门禁。
- 与本次 CI 健全度 grill 的其余产出(uc node 测试进 CI、跨包覆盖率可见、`-race`)同分支落地。
