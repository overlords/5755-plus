# SDK 屏幕方向适配收口为「开屏即定」,不做旋转实时切换

设计系统并入的 07 §1.12 原承诺「SDK 层监听配置变化、旋转时实时切换形态」。2026-06-13 实测核对发现:SDK 全域**无任何配置变化监听器**;实际适配是——弹性弹窗(`MATCH_PARENT`/`WRAP`/weight)靠 Android 在宿主 `configChanges` 下自动 re-measure/layout 随旋转重排(协议、登录窗口已实测正常),而固定 dp 弹窗(小号选择面板)与方向分支弹窗(支付抽屉右↔底、用户中心 80% 宽)只在**挂载时**按当前方向选一次形态。即实为「开屏即定」,非「旋转实时切换」。

决定:**收口为「开屏即定」**——SDK 在弹窗挂载时读一次方向选定形态,不注册 `ComponentCallbacks`、不做旋转实时 re-layout。弹性弹窗随 Android 自动重排;固定/方向分支弹窗旋转时保留挂载时的形态。

理由:
- 接入游戏几乎都**锁定方向**(锁横或锁竖),开着 SDK 弹窗再旋转极罕见。
- 带状态保留的实时 re-layout 成本高:固定 dp 要重算、方向分支(支付抽屉右↔底)是**结构性切换**、且须保留输入/勾选/倒计时态。对一个边角场景不划算。
- 弹性弹窗(协议/登录/维护/状态)已靠 Android 自动 relayout **免费**获得旋转重排(实测正常),无需 SDK 介入。

被否方案:
- 按原 §1.12 实现 `ComponentCallbacks` 监听器 + 旋转实时切换:完整但成本不匹配收益;真有会旋转的接入游戏再补。
- 把固定 dp 弹窗全改弹性布局靠自动 relayout:小号选择面板的 weighted body 依赖固定高,改弹性有难度;且方向分支(抽屉右↔底)仍无法靠自动 relayout 切换。

推论:
- §1.12 已改写:删除「SDK 监听配置变化」的错误描述,明确「开屏即定」+ 已知边界。
- 宿主 `android:configChanges` 声明(01 §6)仍**必需**,但作用是「旋转时不重建 Activity、不销毁 SDK 浮层、保留业务态」,不是「SDK 重布局」;未声明则旋转重建 Activity、浮层与业务态一并丢失。
- 横屏「开屏即定」形态须纳入回归/验收覆盖:弹性弹窗(协议/登录)已实测;固定 dp(小号选择面板)与方向分支(支付抽屉、用户中心)的横屏开屏形态**尚未验证**(受 realname 中文输入限制未能 adb 直验),需补横屏用例或手动横屏走查。
- 将来出现允许自由旋转的接入游戏时,补 `ComponentCallbacks` 监听器实现实时切换,§1.12 与本 ADR 随之修订。
