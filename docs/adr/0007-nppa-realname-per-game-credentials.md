# ADR-0007:NPPA 防沉迷实名核验——平台非运营主体,凭据按接入游戏(per-game)由接入者授权

状态:已接受(2026-06-13)

## 背景

v2 实名一直是 mock(身份证格式 + 出生日期算年龄)/ 生产 fail-closed,从未真接。决定接入**国家新闻出版署「网络游戏防沉迷实名认证系统」(NPPA)** 的官方接口(规范 V2.1)做真实身份核验。

NPPA 三个接口天然分两块:
- **身份核验**:(一)实名认证 `check` + (二)结果查询 `query` → `status`(0成功/1认证中/2失败)+ `pi`(38位用户唯一标识,前6位编码出生日期)。
- **防沉迷强制**:(三)行为数据上报 `loginout` → 跨游戏统计未成年时长、强制时段。

NPPA 凭据三件:`appId`(NPPA 发放)、`bizId`(**= 游戏版号备案识别码**)、`secretKey`(AES/签名密钥)。其中 `bizId` 与具体游戏的版号一一对应,`appId`/`secretKey` 由 NPPA 发给**已备案的游戏运营方**。

## 决策

1. **范围限定**:v2 只接**身份核验**(一+二),把实名 mock 换成 NPPA 真认证 + 权威 `pi`/年龄,门禁模型不变(未实名阻断、未成年阻支付)。**行为上报(三)+ 未成年时段/时长强制**维持 `04 §340` 的 v2 范围外,作为独立合规里程碑后续立项(届时改 04/01)。

2. **凭据 per-game,由接入者授权**:5755 是 SDK / 分发平台,**不是 NPPA 运营主体**;每个接入游戏的版号、NPPA 接入资格归游戏运营方。因此 NPPA 凭据(`appId`/`bizId`/`secretKey`)**按 `game_id` 存储,由接入者(游戏运营方)备案后提供**。平台用游戏 X 的凭据为游戏 X 的用户调用 NPPA。**未配置 NPPA 凭据的游戏**:生产实名 fail-closed(无法核验),dev 走 mock。

3. **密码学按官方向量钉死**:请求体 `AES-128-GCM + BASE64`,签名 `SHA256(secretKey + 字典序 Key-Value 拼接 + body)`。用规范示例(secretKey + 明文 + 密文 + sign)做黄金向量验证,不盲写。

## 后果

- `games`(或独立表)增加 NPPA 凭据列;接入流程要求接入者提供 NPPA 凭据(自助配置属未来 console 范围)。
- **不存在单一全局 NPPA 密钥**——与短信(平台一套全局 JDCloud)相反。
- **真实端到端核验推迟**到首个接入者授权 `appId` 后;在此之前实名生产 fail-closed,但客户端 + 密码学 + per-game 管道已就绪(密码学经官方向量验证)。
- 异步"认证中"(status=1)需懒查询机制(提交存 ai 待查,48h 内经查询接口拿结果)。

## 被否决的替代

- **平台作为 NPPA 运营主体、用一套全局 appId**:否决——`bizId` 必须等于各游戏版号,平台不持有接入游戏的版号,全局单一凭据无法满足 NPPA 的版号绑定,且不合规。
