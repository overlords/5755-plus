# 用户中心走独立「平台对玩家」API 面,不复用 SDK 网关面

用户中心是平台远程 H5 单页(uc.xingninghuyu.com,以下「uc SPA」),**以主账户为核心**:需要主账户身份、账号安全(换绑手机/修改密码)、订单、实名状态等数据。这些数据由谁、以何种方式提供,是本决定要回答的问题。

## 背景与约束

- `/api/sdk/v2/*`(04 契约)是 **SDK 网关面**:入站强制 **HMAC-SHA256 验签**(签名密钥按渠道/游戏内嵌进 AAR,见 `signature.Middleware`),且是**游戏小号(account)口径**——不返回主账户身份、账号安全、订单。
- uc SPA 跑在浏览器里,**拿不到内嵌签名密钥**,无法复刻 HMAC;且它要的数据形状与网关面完全不同。
- 因此 uc SPA **不可能复用 `/api/sdk/v2`**。
- ADR-0003 把服务端范围定为「只含 SDK 网关面」。本决定突破这一边界,故立档。
- 「API 在本仓库」是既定决定(用户中心前端在 uc.xingninghuyu.com,但其数据 API 落本仓库 Go 服务端)。

## 决定

在**同一 Go 服务端**新增一套与网关面**并列**的「平台对玩家」API 面:

- 路径前缀 `/api/uc/v2/*`(`uc` = user center;与网关面 `/api/sdk/v2` 并列,不混用)。
- 鉴权:**`platformToken` Bearer**,复用 04 已定义的 `X-M5755-Platform-Token` 请求头;**不走 HMAC**。会话仍绑 `gameId` 三元组校验(与网关面会话语义一致)。
- 来源受控:`origin` 校验 + CORS 白名单(只允许配置的 `userCenterUrl` 域),防任意页面调用。
- 失效口径:主账户失效统一 `reason=platform_account_invalid`(沿用 04 枚举),uc SPA 据此调 `postAccountAction("session_invalid")` 收口(06 §4),不新增 bridge 方法。
- 生产防护:与网关面一致的生产构建约束;`/api/uc/v2` 不复用 dev 控制面注入路径。

## 影响

- 服务端范围从「单一 SDK 网关面」扩为**「SDK 网关面 + 用户中心面」两面**;两面鉴权机制不同(HMAC vs Bearer),需分中间件、分包(`internal/api` 下区隔)。
- 04 仍是网关面唯一口径;用户中心面另立口径文档(见 `docs/06a-user-center-h5-page.md`),不塞进 04。
- CORS / origin 校验、Bearer 中间件、主账户数据模型(订单、绑定手机、实名状态)为净新增工作。

## 否决的替代

- **复用 `/api/sdk/v2`**:浏览器无法 HMAC 签名,数据口径也不对——技术上不可行。
- **uc API 单独起服务/仓库**:与「API 在本仓库」决定冲突,且徒增部署面。
- **维持 ADR-0003 不动、用户中心不提供动态数据**:退回内置最小回退页(仅切换小号/退出登录两行,见 06 §11.2 未配置 URL 分支),放弃账号安全/订单/实名——不满足「以主账户为核心」的产品口径。
