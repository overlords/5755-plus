# v2 在本仓库内从零重写,不搬运旧实现

旧项目(`~/Developer/m5755`,Gradle 工程,纯 Java)有一套能跑通模拟器验收的实现,但带有 4 个已核实缺口(08 文档 B2-B5,其中两个 P1)和未知数量的文档外漂移。我们决定:v2 代码在本仓库内开发(文档 + 代码同仓),并且**从零重写**——旧代码只作阅读参考,不整体复制任何文件。

理由:07 UI 规格本就自旧实现逐行提取,照文档重写 UI 等价于照旧代码重写;业务状态机与平台网关恰是 B2-B5 所在地,必须重写;基础件(轻量网络/键值存储)边界已由文档规定,重写成本低。整体或逐模块搬运会把未记录的漂移带入,使"以文档为唯一口径"退化为"以旧代码为事实口径",并产生"哪些是搬的、哪些是写的"的混合谱系。重写的隐性收益:每一行代码都被迫对照文档,等于在实现期对 8 份文档做完整性压力测试。

## Considered Options

- 整体搬运旧工程后修缺口——起步快,但继承漂移,口径倒挂。
- 逐模块评审后选择性搬运——引入混合谱系,省下的工夫不抵评审与甄别成本。
- 代码放独立仓库——跨仓对照规格,文档/代码漂移风险回归(旧项目 23 份文档失控的根因)。
