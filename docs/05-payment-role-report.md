# 05 支付与角色上报

本文是 5755 Android SDK 最小化 AAR 角色上报与支付链路的权威业务规格,规定角色上报的触发时机与字段语义、支付流程的前置条件与订单入参规则、客户端支付回调与充值回调的责任划分,以及防沉迷支付门禁的阻断语义。SDK 实现、业务配套 UI、诊断与回归均以本文为准。

分工边界:本文只定义业务层规格。角色上报(`PUT /api/sdk/v2/roles`)、支付创建(`POST /api/sdk/v2/orders`)等端点的请求/响应字段表、签名与传输约定、充值回调(Payment Callback)的协议字段,以《04 平台网关 API》为准,本文不重复协议字段表。术语口径以《02 术语表》为准。

## 1. 角色上报

### 1.1 触发时机

角色上报由接入方显式调用,固定发生在以下条件全部满足之后:

1. 游戏服务端登录态校验通过(SDK 已返回当前游戏小号 ID 与登录令牌,且游戏服务端完成校验);
2. 游戏进入已完成;
3. 游戏已拥有该当前游戏小号在当前区服下的角色资料。

固定规则:

- **SDK 不自动上报角色。** 小号返回后、登录成功后,SDK 都不得代替接入方触发角色上报。
- **游戏无法区分唯一角色时,不调用角色上报。** 角色只有在游戏能够区分唯一角色时才构成可上报对象;接入方不得为满足调用而构造虚假的角色数据模型。
- 角色上报是登录态归属敏感动作:SDK 在发起上报前做本地已登录且登录令牌非空的前置检查,未登录或令牌为空时直接失败并输出诊断,不发起平台请求。

### 1.2 RoleInfo 字段语义

角色上报通过公开对象 `RoleInfo` 与角色上报 API(`sendRoleInfo`)完成。所有字段必填;**确实不存在的字段填 `"-1"`**,但该规则不适用于 `roleId`。

| 字段 | 语义 | 缺失口径 |
| --- | --- | --- |
| `serverId` | 区服 ID,角色所在游戏服务器/服区/分片的标识 | 无区服概念时传 `"-1"` |
| `serverName` | 区服名称 | 无区服概念时传 `"-1"` |
| `roleId` | 游戏内唯一角色 ID | **不允许 `"-1"`**;无法提供唯一角色 ID 即不调用上报 |
| `roleName` | 角色名称 | 不允许为空 |
| `roleLevel` | 角色等级 | 不允许为空 |
| `roleCE` | 角色战力 | 无战力概念时传 `"-1"` |
| `roleStage` | 角色关卡/进度 | 无关卡概念时传 `"-1"` |
| `roleRechargeAmount` | 角色累计充值金额 | 必须是 `"-1"` 或保留 2 位小数的金额字符串(如 `"328.00"`) |
| `roleGuild` | 公会 ID | 无公会概念时传 `"-1"` |

### 1.3 客户端校验规则

SDK 在发起上报请求前对 `RoleInfo` 做客户端校验,校验失败时上报直接失败并输出可采集诊断,不发起平台请求:

- 任一字段为空,失败;
- `roleId` 为空或为 `"-1"`,失败;
- `roleRechargeAmount` 不是 `"-1"` 且不符合保留 2 位小数的金额格式,失败。

诊断输出必须能够区分以下失败原因:字段缺失/非法、未登录或登录令牌为空、网络失败、平台失败。渠道异常不阻断角色上报(渠道回退 `default` 后链路继续)。

### 1.4 用途

角色资料用于三类平台侧业务,接入方应据此理解字段质量要求:

- **运营统计**:区服、角色、等级、战力、关卡、累计充值和公会数据进入平台运营统计,字段必须规范化、可信;
- **活动计算**:平台活动以上报的角色资料为计算输入;
- **支付归属**:角色资料与订单中的区服角色字段共同构成支付归属(订单在平台侧归属于哪个当前游戏小号、区服和角色)的业务关联基础。

## 2. 支付流程

### 2.1 前置条件

支付由玩家在游戏内触发购买行为后,由游戏基于已获取的当前游戏小号和登录令牌发起(游戏侧支付触发)。业务前置条件为:已登录(持有当前游戏小号与登录令牌)、游戏进入已完成、防沉迷支付门禁放行。

SDK 本地校验范围与业务前置的关系:

- SDK 不强制判断游戏是否已完成游戏进入——是否进入游戏由接入方流程保证;
- SDK 只校验:SDK 本地登录态、当前游戏小号、登录令牌非空、防沉迷支付门禁、支付订单入参完整性;
- 任一本地校验失败时支付直接失败并输出诊断,不发起订单创建请求。

### 2.2 Order 对象字段语义

支付通过公开对象 `Order` 与支付 API(`recharge`)完成。订单是玩家为游戏商品发起的支付请求,必须显式绑定当前游戏小号、区服、角色、金额、商品和 CP 订单号,使支付归属明确。

| 字段 | 语义 | 客户端校验 |
| --- | --- | --- |
| `amount`(金额) | 订单金额,单位元 | 必须大于 0,不能是 NaN/Infinity |
| `cpOrderId`(CP 订单号) | 游戏服务端为本笔订单生成的唯一订单标识 | 不能为空,长度不超过 128 |
| `commodity`(商品名称) | 本次购买的商品 | 不能为空 |
| `serverId` | 支付区服 ID | 不能为空 |
| `serverName` | 支付服务器名称 | 不能为空 |
| `roleId` | 支付角色 ID | 不能为空 |
| `roleName` | 支付角色名称 | 不能为空 |
| `roleLevel` | 支付角色等级 | 不能为空 |

区服角色字段与 `RoleInfo` 共享角色归属字段基类(`RoleMeta`),语义一致。协议层请求字段表见《04 平台网关 API》支付创建端点。

### 2.3 订单入参唯一来源:禁止演示订单兜底

**支付 UI 展示和支付请求必须完全取自 `Order` 入参。** 这是上线阻断级规格:

- 金额、商品名、区服、角色、CP 订单号等支付归属字段缺失或非法时,支付必须失败并输出诊断;
- 禁止演示订单兜底:SDK 在缺少真实订单字段时,不得使用固定金额、商品、区服、角色或 CP 订单号继续支付,不得做字段默认值或本地补齐;
- 支付确认 UI 上展示的金额、商品、区服、角色信息必须与 `Order` 入参逐字段一致。

### 2.4 CP 订单号

- CP 订单号由**游戏服务端**在调用支付前为一笔订单生成,游戏服务端负责其唯一性,用于识别重复或重放的支付流程;
- SDK 不生成、不改写 CP 订单号,只透传并校验非空与长度;
- 5755 平台在充值回调中原样返回 CP 订单号,游戏服务端据此完成订单归属和金额校验。

### 2.5 SDK 自有支付容器

支付流程的展示与跳转由 SDK 自有支付容器承载:

- 最小化 AAR **不引入支付宝、微信等第三方支付原生 SDK**,不保留对应的 Manifest 组件、queries 与依赖;
- SDK 自有支付容器只承载支付流程展示和跳转(承接平台返回的支付入口),实际资金流转由平台与第三方支付渠道在 SDK 之外完成;
- 订单创建、支付入口下发遵循服务端订单/回调协议(见《04 平台网关 API》),SDK 不在客户端构造支付结果。

## 3. 两类回调的责任划分

客户端支付回调与充值回调是两条独立链路,必须在命名、展示、回调和文档中强制分开。这是支付链路最容易出错的边界,以下口径为强制规格。

### 3.1 客户端支付回调

客户端支付回调是 SDK 客户端在支付流程结束或进入处理中状态时,向接入方客户端通知客户端支付状态的回调。

- **方向**:SDK 客户端 → 游戏客户端;
- **语义**:仅描述本次客户端支付流程的进展,口径收敛为三类:
  - **已交接**——客户端支付流程已结束并交回游戏,等待游戏服务端处理充值回调;
  - **处理中**——支付仍在进行或结果未明,接入方可提示玩家等待;
  - **未完成**——支付被取消或失败,本次客户端流程终止;
- **用途**:只能用于界面提示和接入方本地流程处理(如关闭支付等待 UI、引导玩家重试);
- **禁止**:客户端支付回调**不是物品发放依据**,不能表示充值回调已经到达或物品已经发放,不改变当前 5755 账户或当前游戏小号状态;SDK UI 与接入方 UI 都不得把任何客户端支付状态命名、展示或回调为"充值回调已确认"或到账结果。SDK UI 可以提示"支付处理中"或"等待服务端充值回调",但仅此而已。

### 3.2 充值回调

充值回调是 5755 平台在确认第三方支付结果后,由**平台服务端通知游戏服务端**的结果消息,是物品发放的**唯一依据**。

- **方向**:5755 平台服务端 → 游戏服务端(不是 Android SDK 端点,协议字段见《04 平台网关 API》);
- **游戏服务端责任**(全部由游戏服务端实现,Android AAR 不内置以下任何一项):
  - 验签:按平台签名规则校验回调签名;
  - 金额校验:回调金额与游戏侧订单金额一致;
  - 账号归属校验:回调中的当前游戏小号、CP 订单号与游戏侧订单记录归属一致;
  - 幂等发放:对同一笔合法充值回调只执行一次物品发放,重复通知只确认处理结果、不重复交付;
  - 充值回调结果响应:成功表示已完成合法校验和物品发放;明确失败表示该笔充值无法成功;其他异常由平台重试;
- **SDK 责任(充值回调支持)**:SDK 在最小上线链路中只提供客户端交接口径、订单归属信息、协议文档说明、模拟回调验收和诊断支持。充值回调支持不表示 AAR 内置游戏服务端的验签、金额校验、账号归属校验或幂等发放实现。

### 3.3 责任矩阵

| 维度 | 客户端支付回调 | 充值回调 |
| --- | --- | --- |
| 方向 | SDK 客户端 → 游戏客户端 | 平台服务端 → 游戏服务端 |
| 内容 | 客户端支付状态(已交接/处理中/未完成) | 第三方支付结果确认 |
| 用途 | UI 提示、本地流程处理 | 验签、金额校验、归属校验、幂等发放 |
| 是否发放依据 | **否** | **是,且是唯一依据** |
| 实现方 | SDK | 游戏服务端(语言/框架不限,按协议实现) |

订单查询接口(`GET /api/sdk/v2/orders/{orderId}`)返回的回调状态仅用于诊断,不影响客户端支付回调的提示语义,更不替代充值回调作为发货依据。

## 4. 防沉迷支付门禁

防沉迷支付门禁是决定当前账户能否发起支付的合规门禁,在支付前置检查中执行。

- **阻断语义**:门禁阻断时,**只失败本次支付并向玩家提示防沉迷限制**;不触发账号变化,不触发登出,不清理当前 5755 账户或当前游戏小号状态。游戏不得把支付门禁阻断误判为账号失效。
- **与进入门禁相互独立**:防沉迷进入游戏门禁与防沉迷支付门禁是两个独立门禁。可进入游戏不代表放行支付;进入门禁阻断只阻断进入流程,支付门禁阻断只失败支付,二者都不触发账号变化或登出。
- **诊断**:门禁阻断必须输出可采集诊断,使客服和支持能够区分"防沉迷限制"与"登录失败/支付失败",解释玩家为何不能支付。
